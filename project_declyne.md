## Session 43 — security uplift kickoff (critical SQLi fix + Drizzle CVE patch)

**Date:** 2026-04-25
**Branch:** claude/research-accounting-software-1K8So
**Phase:** session 1 of 6 in the security-uplift series. Lands on the feature
branch; one PR to main lands at the end of session 48.

**What shipped:**

- New 5-step session-end ritual locked into CLAUDE.md (memory → CLAUDE.md →
  debug pass → commit → push). New "Session scoping" preamble added above the
  ritual block. Push during sessions 43-48 targets
  `claude/research-accounting-software-1K8So`; from session 49+ it returns to
  `main`.
- **Critical SQL-injection fix in `apps/worker/src/routes/debts.ts`.** The old
  PATCH handler built `setClause` from raw `Object.keys(patch)` with no
  allowlist (lines 67-70 pre-fix), so any caller with a valid `API_TOKEN`
  could send a payload like
  `{"principal_cents = 0; DROP TABLE transactions; --": 1}` and execute
  arbitrary SQL. Fixed by introducing a new `parseDebtPatch(raw)` helper
  (mirrors `parseGoalPatch`/`parseHoldingPatch`) that validates each
  documented field and silently drops any other key. The PATCH handler now
  iterates a fixed `ALLOWED_PATCH_FIELDS` array, never `Object.keys(patch)`.
  Per-field validation: name (≤120 chars, non-empty), principal_cents and
  min_payment_value (0..1e11), interest_rate_bps (0..50000),
  min_payment_type ('fixed'|'percent'), statement_date and payment_due_date
  (1..31), account_id_linked (string|null with empty-string-as-clear),
  archived (truthy → 1).
- **Defense-in-depth tightening of `apps/worker/src/routes/allocations.ts`
  PATCH.** Was safe-by-construction (parseAllocPatch already narrowed keys)
  but used the same `Object.entries(patch)` shape as the bad debts pattern.
  Replaced with explicit `ALLOC_PATCH_FIELDS` array iteration so future
  contributors can't accidentally introduce a regression by adding a key to
  AllocPatch without updating the SET writer.
- **Drizzle CVE patched.** `drizzle-orm` ^0.38.2 → ^0.45.2 (0.45.2 is the
  current `latest` dist-tag and the documented fix for the SQL-identifier
  escape CVE). `drizzle-kit` ^0.30.1 → ^0.31.10. Only one drizzle-orm import
  in the codebase (`sqliteTable, text, integer, primaryKey` from
  `drizzle-orm/sqlite-core`) — schema typings only, no query builder usage —
  so the bump is low-risk and required no migration.
- **Eleven new tests** asserting attacker-controlled keys are silently
  dropped from both `parseDebtPatch` and `parseAllocPatch`. The defense-test
  payload includes a classic SQLi shape, a pivot-to-other-tables shape, and
  a `__proto__` prototype-pollution attempt; all are dropped, only the
  legitimate field survives.
